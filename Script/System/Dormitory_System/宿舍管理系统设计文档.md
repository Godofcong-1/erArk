# 宿舍管理系统设计文档

## 1. 文档目的与范围

本设计文档用于规划“宿舍管理系统”的数据、面板、行为与结算接入方案。目标是基于现有基建系统架构（含发电能源管理系统）进行最小侵入扩展，实现以下两项核心功能：

1. 管理干员所属宿舍。
2. 调整各层宿舍管理员。

并配套完成：

1. 新增职业/工作类型“宿舍管理员”。
2. 宿舍管理员对同层干员睡眠恢复效率的加成。
3. 宿舍管理员的两阶段循环日常行动（阶段0：到岗整理，阶段1：移动处理）。

## 4. 功能设计

## 4.1 宿舍管理面板（总入口）

### 4.1.1 面板定位

在基建管理系统中新增“宿舍管理系统”入口，进入后统一承载两项功能：

1. 管理干员所属宿舍。
2. 调整宿舍管理员。

### 4.1.2 建议 UI 结构

标题：宿舍管理系统

Tab 页：

1. 宿舍总览
2. 管理干员所属宿舍
3. 调整宿舍管理员

宿舍总览内容：

1. 宿舍区块等级（来自 `cache.rhodes_island.facility_level[4]`）。
2. 当前开放层数/房间数（由 `facility_open` 推导）。
3. 每层管理员概览（已任命/未任命）。
4. 恢复效率说明（基础效率 + 管理员加成）。

交互规范：

1. 统一采用 `draw` 系列组件构建按钮和文本。
2. 统一通过 `flow_handle.askfor_all(return_list)` 获取输入。
3. 每页均提供返回上级按钮，保持与现有基建面板一致。

## 4.2 管理干员所属宿舍

### 4.2.1 业务目标

允许玩家手动调整干员的宿舍归属，例如将干员从 A 宿舍移动到 B 宿舍。

### 4.2.2 关键字段

干员宿舍相关字段：

1. `character_data.dormitory`：当前宿舍位置。
2. `character_data.pre_dormitory`：临时变更前宿舍记录（用于助理同居/卸任恢复）。

### 4.2.3 与助理逻辑的兼容规则

必须遵守以下规则，避免破坏现有助理系统：

1. 助理同居开启时：先 `pre_dormitory = dormitory`，再迁入博士房间。
2. 助理同居关闭或卸任恢复时：从 `pre_dormitory` 恢复原宿舍。
3. 手动换宿舍功能不得随意覆盖正在用于“临时恢复”的 `pre_dormitory`。

### 4.2.4 建议约束

建议在执行手动换宿舍时增加前提判断：

1. 助理同居激活中的角色不可直接换宿舍。
2. 监禁/特殊状态角色不可直接换宿舍。
3. 目标宿舍需开放且未满员。

### 4.2.5 操作流程

1. 选择目标干员。
2. 展示可迁入宿舍列表（同层/跨层均可，取决于设计策略）。
3. 校验目标可用性（开放、容量、前提条件）。
4. 更新 `dormitory` 并记录日志。
5. 刷新面板显示。

## 4.4 调整宿舍管理员（每层一名）

### 4.4.1 层的定义

宿舍“层”按宿舍区处理（1 区到 9 区），每区视为一层管理单元。

### 4.4.2 开放层判定来源

开放范围由基建设施等级决定：

1. 宿舍区设施等级取 `cache.rhodes_island.facility_level[4]`。
2. 通过设施效果配置映射到可开放区间。
3. 最终由 `cache.rhodes_island.facility_open` 标记具体房间开放状态。

地图来源：

1. `data/map/宿舍/` 下各区地图。
2. 每区含管理员房间（舍管房）用于管理员到岗点。

### 4.4.3 数据结构建议

在 `Rhodes_Island` 新增：

```python
# 输入类型: 无
# 输出类型: Dict[int, int]
# 功能: 记录每层宿舍管理员，key=层号(1-9), value=角色id(0表示未任命)
dormitory_managers: Dict[int, int]
```

初始化建议：

1. `get_base_zero()` 中为 1~9 层初始化为 0。
2. 读档时缺字段自动补齐，兼容旧存档。

### 4.4.4 任命流程

1. 玩家选择层号。
2. 校验该层是否已开放。
3. 展示候选干员列表。
4. 任命后写入 `dormitory_managers[layer] = character_id`。
5. 若已有管理员，先卸任再替换。

## 4.5 宿舍管理员效果：睡眠恢复效率加成

### 4.5.1 现有恢复入口

睡眠恢复结算位于 `Script/Settle/realtime_settle.py` 的 `settle_sleep()`。

当前恢复已叠加宿舍设施效率（`basement.calc_facility_efficiency(4)`）。

### 4.5.2 新增加成原则

在“非监禁睡眠”分支增加“同层宿舍管理员加成”：

1. 根据睡眠角色所属层定位该层管理员。
2. 读取管理员学识能力等级。
3. 按学识等级换算效率倍率，叠乘到现有 `final_adjust`。

示例公式（可调）：

$$
final\_adjust = final\_adjust \times (1 + knowledge\_level \times 0.05)
$$

### 4.5.3 学识读取

学识能力来自角色能力字典（能力 ID 以现有配置为准），通过管理员角色的 `ability` 字段读取。

## 4.6 宿舍管理员日常行动设计

### 4.6.1 行为目标

宿舍管理员在工作时间内基于 `#sym:dormitory_admin_phase` 执行两阶段循环：

1. 阶段 0（到岗整理）：优先到宿舍管理员室，并在室内执行整理宿舍意见。
2. 阶段 1（移动处理）：前往本轮问题目标宿舍，并在目标宿舍处理宿舍问题。
3. 每轮处理完成后由状态机将阶段切回 0，进入下一轮循环。

### 4.6.2 状态机设计

宿舍管理员阶段从原多状态简化为 2 个阶段值：

1. `dormitory_manager_action_0`：到岗整理阶段。
2. `dormitory_manager_action_1`：移动处理阶段。

AI 目标链路固定使用 `data/target/default/target.csv` 的 351~355：

1. 351：到岗时间 + 阶段 0 + 不在宿舍管理员室 -> 去宿舍管理员室。
2. 352：工作时间 + 阶段 0 + 不在宿舍管理员室 -> 去宿舍管理员室。
3. 353：工作时间 + 阶段 0 + 在宿舍管理员室 -> 整理宿舍意见。
4. 354：工作时间 + 阶段 1 + 不在问题目标宿舍 -> 移动至目标宿舍。
5. 355：工作时间 + 阶段 1 + 在问题目标宿舍 -> 处理宿舍问题。

其中 353/355 为执行态（type=22），326/328 为移动态（type=21）。

### 4.6.3 关键筛选规则

随机目标宿舍筛选时应满足：

1. 属于管理员所属层。
2. 房间已开放。
3. 可达（地图可进入）。
4. 过滤不应进入场景（如特殊状态占用）。

### 4.6.4 最小接入点建议

1. 在 WorkType/target 配置中声明宿舍管理员自动行为链，并绑定 351~355。
2. 在 NPC 自动工作逻辑中仅维护两阶段切换（阶段 0 <-> 阶段 1）。
3. 在行为结算层挂接“整理意见/处理问题”的效果，并在处理完成后回写下一轮阶段。

## 6. 存档兼容与风险控制

## 6.1 兼容策略

新增字段（如 `dormitory_managers`）必须在读档时提供默认值，避免旧档加载异常。

## 6.2 主要风险

1. 覆盖 `pre_dormitory` 导致助理卸任后无法恢复原宿舍。
2. 层判定错误导致可任命层与真实开放层不一致。
3. 状态机中断导致管理员卡在中间状态。
4. 睡眠加成重复叠加或作用到非同层角色。

## 6.3 测试清单

1. 任命/撤换各层管理员与 UI 展示一致。
2. 助理同居 + 手动换宿舍互斥逻辑正确。
3. 宿舍升级后可任命层数同步增加。
4. 睡眠恢复加成仅作用于同层且随学识变化。
5. 管理员按 30 分钟 + 10 分钟循环稳定执行。
